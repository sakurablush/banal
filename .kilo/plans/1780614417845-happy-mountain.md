## 1. Executive Summary

Banal AI's core value proposition is **"verified free AI tools — no bullshit."** This plan establishes a systematic, repeatable process to make that claim provably true. Current state: 200+ tools in `zero-key-tools.ts` with rich metadata (caveats, quality notes, access types). Gap: No automated verification that tools still exist, URLs resolve, descriptions match reality, or caveats are current. This plan creates a **continuous trust pipeline** — automated link checks, content freshness audits, transparency scoring, and user-facing trust signals — so every visitor can see _why_ they should trust each entry.

**Key recommendations:**

1. **Automated link/health monitoring** (weekly CI) — catch dead/changed tools before users do
2. **Transparency Score per tool** — visible badge showing verification status, last checked, caveats completeness
3. **User-facing "Trust Dashboard"** — aggregate stats: % tools verified this week, % with caveats, % no-login
4. **Safety/Education layer** — contextual warnings (privacy, data residency, commercial use limits) + "How to choose" guides
5. **Community correction loop** — "Report issue" on every tool card feeding a triage queue

**Top risks:** Tool churn (APIs shut down, free tiers shrink), manual curation bottleneck, false sense of security if automation has blind spots.

## 2. Objectives and Scope

### Business & Technical Goals

- **Trustworthiness:** Every tool entry is verifiably accurate (URL works, name matches, description reflects current reality)
- **Transparency:** Caveats, limits, and trade-offs are prominent, standardized, and machine-readable
- **Education:** Users understand _how to evaluate_ tools, not just _which_ tools exist
- **Safety:** Privacy risks, data residency, and commercial-use traps are flagged before the user clicks

### In Scope

- Automated verification pipeline (link checks, content diffs, accessibility checks)
- Tool metadata schema enhancements (verification timestamps, transparency scores, safety tags)
- UI components: Trust badges, Trust Dashboard, Safety warnings, Report Issue flow
- Curation workflow: Triage queue, reviewer guidelines, changelog
- Documentation: "How we verify", "How to read entries", "Safety guide"

### Out of Scope

- Adding new tools (separate curation process)
- Backend infrastructure (static site stays static)
- Paid/affiliate tooling (explicitly anti-goal)
- User accounts or personalization

### Key Assumptions

- Site remains static (GitHub Pages / Cloudflare Pages) — verification runs in CI, results committed as JSON
- Tool churn rate: ~5-10% per quarter (based on 2024-2025 AI tool landscape observations)
- Volunteer curation capacity: 1-2 hours/week for triage
- Users value "last verified" timestamps over "last updated" timestamps

## 3. Threat Model and Risk Analysis

### Assets & Trust Boundaries

| Asset                   | Boundary                      | Threat if Compromised                                |
| ----------------------- | ----------------------------- | ---------------------------------------------------- |
| Tool directory accuracy | `zero-key-tools.ts` → UI      | User wastes time on dead tools; loses trust in brand |
| Caveat completeness     | Metadata → rendered badges    | User hits unexpected paywall/limit; privacy exposure |
| Verification timestamps | CI artifact → UI              | Stale "verified" badge creates false confidence      |
| Community reports       | GitHub Issues / form → triage | Spam/fake reports waste maintainer time              |
| Safety tags             | Metadata → UI warnings        | User sends sensitive data to non-private tool        |

### STRIDE Analysis

| Threat                                          | Vector                                  | Likelihood        | Impact                | Mitigation                                                                |
| ----------------------------------------------- | --------------------------------------- | ----------------- | --------------------- | ------------------------------------------------------------------------- |
| **Spoofing**: Fake tool entry added             | PR with malicious tool                  | Low (code review) | High (malware link)   | Required fields + automated URL reputation check (VirusTotal/GSB)         |
| **Tampering**: Caveat removed silently          | Direct edit to `zero-key-tools.ts`      | Medium            | High (hidden paywall) | Schema validation + diff alert on caveat field changes                    |
| **Repudiation**: "I never saw that warning"     | User claims no warning shown            | Medium            | Medium (liability)    | Signed verification artifacts; UI shows "Verified YYYY-MM-DD"             |
| **Info Disclosure**: Tool collects PII silently | Tool changes TOS post-verification      | High (common)     | High                  | Weekly re-check of TOS/privacy policy URLs; "Privacy policy changed" flag |
| **DoS**: Verification pipeline fails            | GitHub Actions quota exhausted          | Low               | Medium (stale data)   | Fallback to cached results; alert on failure                              |
| **Elevation**: Community report injects XSS     | Report description rendered unsanitized | Low               | High                  | Sanitize all user input; CSP headers                                      |

### Attack Trees (Critical Paths)

**Path 1: User trusts dead tool → wastes time → loses trust**

- Root: Tool URL returns 404/redirects to paywall
- Mitigation: Weekly HEAD/GET check; `lastVerified` timestamp; "Report broken" button

**Path 2: User sends sensitive data to non-private tool → privacy breach**

- Root: Tool marketed as "private" but sends data to server
- Mitigation: `access: 'open-source'`/`'self-host'`/`'no-login'` distinction; privacy badge; data residency tag

**Path 3: Maintainer misses caveat regression → user hits surprise limit**

- Root: Tool changes free tier; caveat not updated
- Mitigation: Diff alert on `caveat`/`qualityNote`/`access` fields; quarterly manual audit of top 50 tools

### Risk Scoring & Prioritized Mitigations

| Risk                                    | Score (L×I) | Priority | Mitigation                                                  |
| --------------------------------------- | ----------- | -------- | ----------------------------------------------------------- |
| Dead/broken links                       | 4×5=20      | P0       | Automated weekly link check + `lastVerified` badge          |
| Caveat drift (hidden paywalls)          | 3×5=15      | P0       | Field-level diff alerts + quarterly top-50 audit            |
| Privacy misrepresentation               | 3×5=15      | P0       | Privacy/safety tags + data residency field + TOS monitoring |
| Stale verification badges               | 3×4=12      | P1       | CI commits `verification-results.json`; UI shows timestamp  |
| Community report spam                   | 2×3=6       | P2       | Honeypot + rate limit + manual triage                       |
| Schema drift (new fields not validated) | 2×3=6       | P2       | Zod schema + CI validation on every PR                      |

## 4. Options Analysis and Recommendations

### 4.1 Verification Pipeline Architecture

| Option                                                | Description                                                                     | Pros                            | Cons                               | Recommendation                     |
| ----------------------------------------------------- | ------------------------------------------------------------------------------- | ------------------------------- | ---------------------------------- | ---------------------------------- |
| **A: GitHub Actions weekly**                          | `node scripts/verify-tools.ts` runs weekly, commits `verification-results.json` | Free, native, versioned results | 6h timeout; no browser rendering   | ✅ **Recommended** — start here    |
| **B: External monitor (Uptime Kuma / Better Uptime)** | Dedicated monitoring service                                                    | Rich dashboards, alerts         | Cost, external dependency          | ⚠️ Phase 2 if scale demands        |
| **C: Client-side verification**                       | Service worker checks links on user visit                                       | Real-time, user-specific        | Privacy concern; no aggregate data | ❌ Not for trust signals           |
| **D: Hybrid (A + community reports)**                 | CI baseline + user "Report issue" buttons                                       | Defense in depth                | More moving parts                  | ✅ **Recommended** — combine A + D |

**Decision:** **Option A + D**. GitHub Actions weekly run (free, versioned) + lightweight "Report issue" button on each tool card posting to GitHub Issues with pre-filled template.

### 4.2 Transparency Score Design

| Option                             | Description                                                               | Pros                                | Cons                                 | Recommendation                      |
| ---------------------------------- | ------------------------------------------------------------------------- | ----------------------------------- | ------------------------------------ | ----------------------------------- |
| **A: Single letter grade (A-F)**   | Simple, familiar                                                          | Instantly scannable                 | Oversimplifies; grade inflation risk | ⚠️ Only as summary                  |
| **B: Multi-dimensional badges**    | Separate badges: Verified, Caveats Complete, No-Login, Privacy-Respecting | Granular; user chooses what matters | More visual clutter                  | ✅ **Recommended** — 4-5 badges max |
| **C: Numeric score (0-100)**       | Weighted formula                                                          | Precise; sortable                   | Opaque formula; gaming risk          | ❌ Avoid                            |
| **D: "Last verified" + checkmark** | Timestamp + green check if all checks pass                                | Honest; no grading                  | No quick "is this good?" signal      | ✅ **Required baseline**            |

**Decision:** **Baseline D (timestamp + checkmark) + Badges B (4 key dimensions)**. Formula: `Verified ✓` (link works) + `Caveats ✓` (caveat field non-empty + qualityNote non-empty) + `No-Login ✓` (access ∈ {no-login, open-source, public-api, self-host}) + `Privacy ✓` (category=security-privacy OR badges include 'private'/'encrypted'/'local').

### 4.3 Safety/Education Layer

| Option                                     | Description                                                            | Pros                      | Cons                                     | Recommendation                                |
| ------------------------------------------ | ---------------------------------------------------------------------- | ------------------------- | ---------------------------------------- | --------------------------------------------- |
| **A: Inline warnings per tool**            | Red banner on tool card: "⚠ Requires account" / "⚠ US data residency"  | Contextual; unavoidable   | Alert fatigue if overused                | ✅ **Recommended** — only for high-risk flags |
| **B: Category-level guide pages**          | `/guides/privacy`, `/guides/choosing-llm`, `/guides/commercial-use`    | Deep education; SEO       | Separate from directory; discoverability | ✅ **Recommended** — 3-4 guide pages          |
| **C: Interactive "Help me choose" wizard** | 3-question flow → filtered recommendations                             | Personalized; engaging    | Maintenance burden; JS complexity        | ⚠️ Phase 2                                    |
| **D: Tooltip glossary**                    | Hover "no-login" → "No account creation required. May still track IP." | Low friction; educational | Mobile hover doesn't exist               | ✅ **Recommended** — tap-to-expand on mobile  |

**Decision:** **A (high-risk inline) + B (guides) + D (glossary)**. Phase 1: glossary + 2 guides (Privacy, Commercial Use). Phase 2: wizard if engagement metrics justify.

### 4.4 Community Correction Loop

| Option                                  | Description                                                          | Pros                        | Cons                              | Recommendation                  |
| --------------------------------------- | -------------------------------------------------------------------- | --------------------------- | --------------------------------- | ------------------------------- |
| **A: GitHub Issues template**           | "Report issue" → pre-filled issue with tool ID, URL, expected/actual | Free; versioned; assignable | Requires GitHub account; friction | ✅ **Recommended** — start here |
| **B: Netlify Form / Formspree**         | No-login form → email/Slack                                          | Zero friction for users     | Spam risk; no public tracking     | ⚠️ Fallback                     |
| **C: Inline "Was this helpful? 👍/👎"** | Quick feedback → aggregated stats                                    | Low friction; quant signal  | No qualitative detail             | ✅ **Add later** — low cost     |
| **D: Discord/Reddit community**         | Discussion forum                                                     | Rich feedback               | Moderation burden                 | ❌ Out of scope                 |

**Decision:** **A (GitHub Issues) + C (helpful votes)**. GitHub Issues for actionable reports; helpful votes for signal on tool quality.

## 5. Detailed Technical Approach

### 5.1 Enhanced Tool Metadata Schema

Extend `ZeroKeyTool` interface in `zero-key-tools.ts`:

```typescript
export interface ZeroKeyTool {
  // ... existing fields ...

  // Verification metadata (populated by CI, committed to repo)
  verification?: {
    lastChecked: string; // ISO 8601
    status: 'ok' | 'redirect' | 'broken' | 'paywall' | 'changed';
    httpStatus?: number;
    finalUrl?: string; // after redirects
    contentHash?: string; // SHA256 of key page content
    checks: {
      link: boolean;
      nameMatch: boolean; // page title contains tool name
      caveatPresent: boolean;
      qualityNotePresent: boolean;
      accessTypeValid: boolean;
    };
    warnings: string[]; // e.g., ["Privacy policy changed", "Free tier reduced"]
  };

  // Safety/privacy tags (curator-maintained)
  safetyTags?: SafetyTag[];

  // Data residency (for privacy-conscious users)
  dataResidency?: 'eu' | 'us' | 'global' | 'unknown' | 'self-hosted';

  // Commercial use clarity
  commercialUse?: 'allowed' | 'restricted' | 'unknown' | 'prohibited';

  // Educational context
  learningCurve?: 'trivial' | 'low' | 'medium' | 'high';
  bestForDetail?: string; // Expanded "bestFor" for guides
}

type SafetyTag =
  | 'no-tracking'
  | 'e2e-encrypted'
  | 'local-first'
  | 'self-hostable'
  | 'open-source'
  | 'audited'
  | 'gdpr-compliant'
  | 'hipaa-ready'
  | 'no-ai-training'
  | 'data-export';
```

### 5.2 Verification Script (`scripts/verify-tools.ts`)

```typescript
// Runs weekly via GitHub Actions
// Input: zeroKeyTools array
// Output: verification-results.json (committed)

interface VerificationResult {
  toolId: string;
  timestamp: string;
  status: 'ok' | 'redirect' | 'broken' | 'paywall' | 'changed';
  httpStatus: number;
  finalUrl: string;
  redirectChain: string[];
  checks: {
    link: boolean;
    nameMatch: boolean;
    caveatPresent: boolean;
    qualityNotePresent: boolean;
    accessTypeValid: boolean;
    tosAccessible: boolean; // Can fetch TOS/privacy policy
  };
  contentHash: string; // For change detection
  warnings: string[];
  diff?: {
    // If contentHash changed
    field: string;
    old: string;
    new: string;
  }[];
}

// Implementation notes:
// - Use undici/fetch with timeout (10s)
// - Follow redirects (max 5), record chain
// - Check for paywall indicators: "upgrade", "pro", "premium", "credit card", "billing"
// - Extract page title + h1 for nameMatch
// - Cache results to avoid re-checking same domain repeatedly
// - Rate limit: 1 req/sec, respect robots.txt
// - Output JSON + markdown summary for PR description
```

### 5.3 CI Pipeline (`.github/workflows/verify-tools.yml`)

```yaml
name: Weekly Tool Verification
on:
  schedule:
    - cron: '0 6 * * 1' # Every Monday 06:00 UTC
  workflow_dispatch: # Manual trigger

jobs:
  verify:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npm run verify:tools # Runs scripts/verify-tools.ts
      - name: Create PR if changes
        uses: peter-evans/create-pull-request@v6
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          branch: tool-verification/update
          title: 'chore: weekly tool verification update'
          body: |
            Automated verification results for ${{ github.event.repository.name }}
            See verification-results.json for details.
          labels: 'automated,verification'
```

## 7. Risks, Assumptions, and Open Questions

### Major Risks & Mitigations

| Risk                                                    | Probability                | Impact   | Mitigation                                                                                          |
| ------------------------------------------------------- | -------------------------- | -------- | --------------------------------------------------------------------------------------------------- |
| **Verification false positives** (paywall not detected) | Medium                     | High     | Multiple detection signals (keywords + DOM patterns + price API); manual review queue for ambiguous |
| **Tool churn outpaces weekly checks**                   | High (AI space moves fast) | Medium   | Add "Report issue" for real-time correction; consider daily checks for top 20                       |
| **Curator burnout** (tagging 200+ tools)                | Medium                     | Medium   | Prioritize top 50; community PRs for tags; Phase 2: ML-assisted tag suggestion                      |
| **GitHub Actions quota exhaustion**                     | Low                        | Medium   | Optimize: parallel checks, cache, skip unchanged domains; self-hosted runner fallback               |
| **Legal liability for bad recommendations**             | Low                        | Critical | Disclaimer on every page; "Not legal/medical/financial advice" in caveats; no warranties            |

### Assumptions That Could Invalidate Plan

1. **Static site constraint holds** — If backend added, verification could be real-time
2. **Tool URLs remain directly accessible** — Some tools move behind Cloudflare/JS challenges
3. **Curator availability** — If 0 hours/week, automation must be 99% sufficient
4. **User trust model** — If users don't care about badges, effort is wasted (measure: click-through on "Report issue")

### Open Questions (Need Decisions Before Phase 1)

1. **Verification result storage:** Commit `verification-results.json` to repo (history, transparency) vs. artifact only (clean repo)? → **Recommend: Commit** — enables "verification changelog" page.

2. **Paywall detection sensitivity:** Aggressive (flag any "upgrade" mention) vs. conservative (only flag hard paywalls)? → **Recommend: Conservative + manual review queue** — avoid false alarms.

3. **Data residency default:** Assume `'unknown'` unless proven? → **Yes** — forces curator to research; `'self-hosted'` for CLI/local tools.

4. **Commercial use default:** `'unknown'` vs `'restricted'`? → **Recommend: `'unknown'`** — safest; curator upgrades to `'allowed'` only with evidence.

5. **Japanese localization:** Translate all new UI strings + glossary + guides? → **Yes, required** — project has EN/JA i18n; budget 30% extra time.

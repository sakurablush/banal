## 1. Executive Summary

Strona Banal AI potrzebuje **ostatecznej metamorfozy wersji produkcyjnej**. Obecnie ma solidną infrastrukturę (200+ narzędzi, TypeScript, wyszukiwanie, i18n EN/JA), ale UX/UI jest półprodukcyjne, testy failing, brakuje particle systemu, a artykuł o Kilo nie jest wpleciony w design. Celem jest **jedna spójna wersja, która po git push wygląda jak final produkt** — crystal clear search-first UI, world-class particle background, skeleton loading, dedupowanych danych, opinii dla użytkownika bez hardware/tokenów, wrappers API do testowania, i full pass testów na zielono.

**Najważniejsze decyzje:**
- Command palette search jako **jedyny główny element** interakcji (Ctrl+K focus, ↑↓ nav, Enter open)
- Particle system jako **atmosfera strony** — nie jako kosmiczny bajer, a jako "ghost in the machine"
- Artykuł o Kilo wpleciony w strukturę strony jako **dowód wiarygodności**, nie jako promotional
- API wrappers jako **"Playground"** — użytkownik testuje darmowe API bez opuszczania strony
- Menu/paginacja hybrydowe: sticky search + filter chips + infinite scroll skeleton

## 3. Threat Model and Risk Analysis

### Assets
- **Narzędzia** — lista 209 zweryfikowanych darmowych narzędzi (najważniejszy asset)
- **Zaufanie użytkownika** — każdy nieprawdziwy/inny tool to utrata wiarygodności
- **Wydajność UI** — search musi być instant, nawet na słabym sprzęcie
- **Dostępność** — EN/JA wersje muszą być 1st class, nie "dodatkowe"

### STRIDE Analysis

| Threat | Risk | Mitigation |
|--------|------|------------|
| **S**poofing: fałszywe narzędzia w liście | HIGH | Każde narzędzie ma `bestFor`, `qualityNote`, `caveat`. Duplikaty usuwane. Testy sprawdzają unikalność ID. |
| **T**ampering: użytkownik edytuje localStorage | LOW | Dane są read-only UI. localStorage tylko dla preferences (lang, last search). |
| **R**epudiation: brak historię zmian narzędzi | MEDIUM | Każdy narzędzie ma jawne pola w TS. Możliwość wersjonowania w przyszłości. |
| **I**nformation disclosure: API keys w URL | HIGH | Zadbamy o NIE pokazywanie kluczy w URL. Playground używa serverless functions z env vars. |
| **D**enial of Service: search laguje | MEDIUM | Debounce 60ms + fuse-search O(n) z early exit. Virtual list jeśli >200. |
| **E**levation of privilege: użytkownik myśli że to AI tool a nie jest | HIGH | Artykuł o Kilo jasno tłumaczy różnicę. Each tool ma `surface` badge (web/api/cli). |

### Attack Trees

**Atak 1: Fałszywe narzędzie w katalogu**
```
Goal: Wprowadzenie nieprawdziwego narzędzia
├── Attack vector 1: copy-paste z marketing
│   └── Mitigation: human review + testy sprawdzające URL format
├── Attack vector 2: narzędzie przestało być darmowe
│   └── Mitigation: caveat pola + manualna weryfikacja co miesiąc
└── Attack vector 3: duplikat ID
    └── Mitigation: test deduplication w CI
```

**Atak 2: Performance degradation na słabym devices**
```
Goal: Użytkownik na starym laptopie nie może używać wyszukiwania
├── Attack vector 1: Particle canvas zbyt ciężki
│   └── Mitigation: requestAnimationFrame + throttle + reduced motion media query
├── Attack vector 2: 209 narzędzi renderowanych naraz
│   └── Mitigation: virtual list / pagination / skeleton loading
└── Attack vector 3: Duży bundle JS
    └── Mitigation: tree-shaking, lazy load particle canvas
```

## 5. Detailed Technical Approach

### Architektura

```
src/
├── api-playground/
│   ├── index.ts              # Main playground component
│   ├── providers.ts          # API provider definitions (free tiers only)
│   └── wrappers/
│       ├── groq-wrapper.ts   # Groq free API
│       ├── huggingface-wrapper.ts  # HF Inference API
│       ├── pollinations-wrapper.ts # Pollinations (image + chat)
│       └── kiloprovider.ts   # Kilo agent (iframe embed if available)
├── particles/
│   └── particle-system.ts    # Canvas particle engine (ghost particles)
├── zero-key-panel.ts         # Command palette (CURRENT — needs finishing)
├── fuse-search.ts            # Already good, keep
├── style.css                 # Needs .zk-* classes + particle styles
├── data/
│   └── zero-key-tools.ts     # Fix remaining duplicates
├── directory.ts              # Keep, minor updates
└── main.ts                   # Wire particles + playground
```

### Komponent: Command Palette (`zero-key-panel.ts`)
```
Stan: {
  query: string
  results: SearchResult[]
  selectedIndex: number
  filters: Set<string>
  isOpen: boolean
}

Flow:
1. Input focused → show 10 recent queries (z localStorage)
2. Typing → debounce 60ms → fuse search → render rows
3. ↑↓ → highlight row (virtual scroll jeśli >100)
4. Enter → open link
5. Esc → clear / blur
6. Ctrl+K → focus from anywhere
```

### Komponent: Particle System (`particles/particle-system.ts`)
- Vanilla JS canvas, no libs needed
- ~150 particles, fade in/out, subtle movement
- Colors: violet/cyan/amber neon palette
- Respects `prefers-reduced-motion`
- Throttled to 30fps on low-end devices
- ZERO dependencies

### Komponent: API Playground (`api-playground/`)
```
Providers config:
{
  id: 'groq',
  name: 'Groq (Free)',
  tier: 'free',
  rateLimit: '30 req/min',
  models: ['llama-3.3-70b', 'mixtral-8x7b'],
  endpoint: 'https://api.groq.com/openai/v1/chat/completions',
  method: 'POST',
  headers: { 'Authorization': 'Bearer $KEY' },
  bodyTemplate: (prompt) => ({...}),
  inputComponent: 'textarea',
  outputFormat: 'markdown'
}
```

Wrapper generuje:
- Formularz wejściowy (textarea + model select)
- Przycisk "Send" → wywołuje endpoint przez **Vite proxy** (żeby ukryć CORS/key)
- Wykres response time + token usage
- Histora zapytań w localStorage
- Kod snippet do skopiowania (curl + JS)

**Bezpieczeństwo:** 
- Proxujemy przez `/api/playground` endpoint na tym samym domain (Vite dev proxy + Cloudflare pages function)
- Nigdy nie eksponujemy kluczy w kliencie
- Możliwość "try without key" → pokazuje error z instrukcją jak dostać key za darmo

### Artykuł o Kilo — struktura na stronie

  <div class="featured-article">

## 7. Risks, Assumptions, and Open Questions

### Risks
1. **Testy mogą być zależne od kolejności** (life filters残留 między testami) — rozwiązane przez `resetZeroKeyPanelFiltersForTests()` ale trzeba uwzględnić w afterEach
2. **Particle system może gryź na słabym Android** — solved przez `prefers-reduced-motion` + throttle
3. **API wrappers mogą wyexponować keys** — solved przez Vite proxy + `?key=` z env
4. **Hugging Face Inference API może mieć CORS** — need to verify przed budową wrappera

### Assumptions
- Użytkownik ma nowoczesną przeglądarkę (Chrome/Firefox/Safari <2 lata)
- Użytkownik ma minimum 1 darmowe API key (lub chce się dowiedzieć jak dostać)
- Strona nie będzie miała więcej niż 500 narzędzi w najbliższych 6 miesiącach

### Open Questions
1. **Czy Kilo ma publiczne API/embed URL** dla iframe w playground? (potrzebuję sprawdzić docs.kilo.ai)
2. **Czy Pollinations API wymaga key** w 2025, czy jest w 100% free?
3. **Czy chcesz Newsletter section** w artykule? (mozna dodac "Get notified when we add tools")
4. **Czy chcemy Polski język** oprócz EN/JA? (user pisze po polsku)
